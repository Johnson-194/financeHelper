\section{Introduction}
This is an individual project made by me in July 2026. The purpose of this project is to make a program that takes user input to record (locally) a history of how much money they are spending, and provide insight into the data collected via graphs and pie charts ranging over selected periods of time. This report aims to cover the entire process of this project, including planning and conclusions.

\section{Analysis}
\subsection{Outline of the Problem}
As a university student, I have used Microsoft Excel to record my finances, but Excel can be difficult to learn alongside my studies, and many issues make it hard to use conveniently, such as money being deposited at different days of the month and not necessarily every month. Many of my peers also find Excel difficult to use and, although useful, not all of them track their finances or know how to budget effectively. I would like to implement a solution to this problem by making it more convenient for students to track their spending, which will hopefully give them insight into their spending habits leading to better budgeting and financial choices, resulting in an improved quality of life for students.
For this program to be convenient, I plan to make a clear user interface where people can interact by entering input and tapping buttons to navigate to other areas of the application. To make the program insightful towards its users, I plan to add graph features where you can select the time frame (beginning to end) and see a graph alongside a summary of their spending with a pie chart. When entering an amount of money, users can also decide to create a category for that item which will be shown in the summary and the pie chart.

\subsubsection{Use of Algorithms}
The use of algorithms here is quite limited, but there are some that may be applicable.
Since the program will mostly be about numbers, we can gather the money spent on each day of the time frame (floats) and sort it to make calculating the median and range of the data easier. \newline
Here are some sorts that I could use. \newline
\textbf{Sorts}:
\begin{itemize}
    \item \textit{Bubble Sort}: Simple but very slow, $O(n^2)$ time complexity on average.
    \item \textit{Insertion Sort}: Better, but still $O(n^2)$ time complexity on average.
    \item \textit{Quick Sort}: Good, $O(nlog(n))$ time complexity on average.
\end{itemize}
Although calculating the statistics don't require searching algorithms as we only need to access the middle, lowest and highest values after sorting, we may need to search for specific dates (the beginning and end) in a data structure that has sorted the data by the date. There are a few examples of what the data structure could be (as it is not yet decided), for instance we could use a binary tree or a list/array.\newline
If the data structure is a list, we would use searching algorithms to find the date. \newline
\textbf{Searches}:
\begin{itemize}
    \item \textit{Linear Search}: Very slow for large data, $O(n)$ on average.
    \item \textit{Binary Search}: Very good, and usable as the data will be sorted by date, $O(log(n))$ on average.
\end{itemize}
If the data structure is a (binary) tree, we can use traversing algorithms to find all of the dates in a time frame, but this requires traversing the entire tree or at least most of it since you can discard some routes that you are sure that no valid dates exist for. The order of adding items to the tree would also vary the efficiency since the tree is unlikely to be "balanced" in which case traversals will no longer discard half the data each time and instead (in the worst case) discard one at a time essentially being turning into a linked list. I also have to make sure that the binary tree is sorted at all times, which shouldn't be an issue as you just have to insert items in the correct place. \newline
\textbf{Traversals}:
\begin{itemize}
    \item \textit{Breadth-First Search}: Records all connected nodes to the first node in a queue (then removes it), adds those to the queue and repeats until the queue is empty.
    \item \textit{Depth-First Search}: Traverse left if possible, if not then traverse right if possible, if not then record current node as visited, backtrack and repeat.
\end{itemize}
Depth-first search doesn't have to go left first, it can try right first and then left if failed. I can make this algorithm more efficient by determining which path to take based on the date I need as the tree will be sorted. I think both algorithms are as useful as each other and one may prove to be better than the other when put into practice but we will see...

\subsubsection{Research into Swift: UI}
In Swift, user interface elements are called \textbf{views}, these can be arranged in certain ways, and there are different types of views available such as \textit{text views}, \textit{image views} and \textit{buttons}. Views can be arranged using \textit{HStacks (horizontally)} and \textit{VStacks (vertically)} and you can also use \textit{NavigationStacks (stack)} and \textit{NavigationSplitViews (multiple columns)} to set how views are presented and transitioned when navigating through the application. All of this is available through the \textbf{SwiftUI} framework, which I will import into the code. Having all of these features available, it will be easy to create a simple, but effective, user interface, enhancing the user experience. Regarding input forms, I can also use a \textit{TextField} (included in SwiftUI) to take in input, although this only allows storing to variables which are volatile (as they are stored in RAM only) they will not save for the next session, so I will need to figure out how to store user input such as the spending data to carry them over between sessions. 

\subsubsection{Research into Swift: Graphs and Pie Charts}
I can use the framework: \textbf{Charts} which allows me to put graphs and pie charts into the application via the \textit{chart} view. This also allows me to colour code the pie, giving each category that the user creates a different colour making it more readable and user friendly. Swift's Charts framework additionally supports line graphs and bar charts making them easier to implement as well. Not only this, but it also allows sections of each bar (in the bar charts) to be colour coded by category. The keys for each colour showing what category they represent is called a \textbf{legend}. Normally, the colours that are used in the charts/graphs is chosen automatically, but you can also choose the colours yourself by using the \textit{chartForegroundStyleScale} modifier and giving it a list of mappings from \textit{name} to \textit{colour} (using the built-in struct: \textbf{Color}). I think it makes more sense to use colours for charts rather than graphs but if it is useful, I will use colours for my graphs when it is implemented.

\subsubsection{Research into Swift: Data Persistence}
The application would not be very useful if the data entered by the user is lost after they close the app, which is exactly what would happen if the program doesn't store the data somewhere other than in variables. So we need data persistence, which means that the data is not lost after the application is closed. Instead, I think the application should find where the data is stored (If there is no data yet, then where it should be stored) and retrieve it when the program starts and it should be the code's responsibility to keep this updated so no data is lost. To implement this feature, I will use the \textbf{SwiftData} framework. The financial data that will be stored is called a \textit{data model} and in order to make it persistent, you annotate the data model with \textit{@Model} (which uses the Model() macro). The data model must be a \textit{class} so I must create a class for the financial data model (individual item and category) which should be straight forward.
